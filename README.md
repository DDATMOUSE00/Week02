# GitHub 브랜치 규칙

우리 팀은 5인 규모의 게임잼 팀이므로, 너무 무거운 프로세스보다는 빠른 협업과 충돌 방지에 집중한다.

## 브랜치 구조

- `main`
  - 최종 결과물만 관리한다.
  - 배포 가능하거나 제출 가능한 안정 버전만 반영한다.

- `dev`
  - 팀원들과 함께 확인된 기능만 머지한다.
  - 작업 중간 결과물이 아니라, 실제로 동작 확인이 끝난 내용만 받는다.

- `features/기능이름/작업자이름`
  - 개인 작업 브랜치는 아래 규칙으로 생성한다.
  - 예시:
    - `features/playerMove/woosung`
    - `features/uiStageSelect/minji`

## 작업 방식

1. 모든 작업은 개인 `features/...` 브랜치에서 진행한다.
2. 작업이 끝나면 바로 `dev`에 머지하지 않는다.
3. 반드시 `dev` 머지 전에 팀원들에게 대면 코드 리뷰를 요청한다.
4. 모든 팀원이 내용을 확인하고 승인한 뒤에만 `dev`로 Pull Request를 머지한다.

## 왜 이렇게 하나요?

이번 게임잼 프로젝트는 하나의 씬에서 작업이 많이 겹칠 수밖에 없다.  
기능 단위로 나누더라도 서로의 작업에 영향이 갈 가능성이 높다.  

그래서 각자 따로 작업하더라도, `dev`에 반영하기 전에는 반드시 함께 확인하고 승인하는 과정을 거친다.

## 요약

- 최종본은 `main`
- 확인된 기능만 `dev`
- 개인 작업은 `features/기능이름/작업자이름`
- `dev` 머지 전 반드시 대면 코드 리뷰
- 팀 전체 승인 후 `dev` 머지
